# ADR-0001: ToolRegistry, ElementSelector, and cross-platform strategy

**Status:** Accepted  
**Date:** 2026-07-02  
**Deciders:** @kurosss  

---

## Context

Проект smith-automation требует архитектурных решений по трём направлениям:

1. **ToolRegistry** — как регистрировать, обнаруживать и выполнять инструменты.
2. **ElementSelector** — как идентифицировать UI-элементы и привязывать их к процессам.
3. **Cross-platform** — как организовать код для поддержки Windows, Linux и macOS.

---

## Decision 1: ToolRegistry — static registration for now

### Option considered: Static registration (chosen)

Инструменты регистрируются в `HashMap<String, Box<dyn Tool>>` вручную в точке входа:

```rust
let mut registry = ToolRegistry::new();
registry.register(ClickTool::new());
registry.register(TypeTool::new());
```

**Pros:**
- Простота реализации на текущем этапе.
- Полный контроль над набором инструментов на этапе компиляции.
- Нет лишних зависимостей (dlopen, libloading).

**Cons:**
- Добавление нового инструмента требует изменения кода ядра.
- Невозможно подключить внешние плагины без перекомпиляции.

### Option considered: Dynamic loading (deferred)

Загрузка инструментов из динамических библиотек (`.dll`/`.so`) через `libloading`.

**Deferred because:**
- Усложняет сборку и распространение.
- Требует стабильного ABI или интерфейса на C.
- Малая потребность на текущем этапе (все инструменты известны).

### Decision

Используем статическую регистрацию. Динамическая загрузка будет рассмотрена при появлении потребности во внешних плагинах.

## Decision 2: ElementSelector with PID binding

UI-элементы идентифицируются через комбинацию:

1. **PID процесса** — ограничивает поиск элементов конкретным приложением.
2. **AutomationId / Name / ControlType** — стандартные UIA-свойства для точной идентификации.
3. **Condition tree** — поддержка составных условий (AND, OR) через UIA `Condition`.

Сам элемент оборачивается в `SafeUIElement` — потокобезопасную обёртку на `Arc<UIElement>` с `unsafe impl Send + Sync`:

- `UIElement` внутри содержит COM-интерфейс, который является free-threaded.
- Все блокирующие COM-вызовы (click, setValue) выносятся в `spawn_blocking`.

**Решение:** `ElementSelector` будет структурой, хранящей PID и опциональные условия поиска. После поиска результат (найденный `SafeUIElement`) сохраняется в `ExecutionContext` для последующего использования инструментами.

## Decision 3: Cross-platform strategy with cfg(windows)

### Levels of isolation

**Level 1 — Dependencies:**

```toml
[target.'cfg(windows)'.dependencies]
uiautomation = "0.25.0"
```

Платформенные крейты подключаются только для соответствующей цели.

**Level 2 — Modules:**

```rust
#[cfg(windows)]
pub mod element;
#[cfg(windows)]
pub use element::SafeUIElement;
```

Модули и публичные типы существуют только на своей платформе.

**Level 3 — Implementation:**

```rust
#[cfg(windows)]
impl SomePlatformTrait for PlatformImpl { ... }
```

Отдельные impl-блоки скрываются за `cfg`.

### Future platform support

Для каждой новой платформы создаётся отдельный крейт:

- `smith-windows` — UIAutomation (существует сейчас).
- `smith-linux` — AT-SPI / X11 (планируется).
- `smith-macos` — Accessibility API (планируется).

Выбор активного backend'а будет через feature flags либо через регистрацию в runtime registry.

### Decision

Весь платформенный код изолируется за `#[cfg(windows)]` на уровне модулей и зависимостей. `smith-core` остаётся полностью платформонезависимым. Новые платформы добавляются как отдельные крейты с тем же интерфейсом `Tool`.

## Consequences

1. **ToolRegistry:**
   - Простота ценой гибкости — инструменты только в составе ядра.
   - Лёгкий рефакторинг при переходе на динамическую загрузку (API registry не изменится).

2. **ElementSelector:**
   - PID-привязка изолирует автоматизацию в рамках одного процесса.
   - SafeUIElement требует `unsafe` — документированы safety invariants.
   - spawn_blocking гарантирует, что async runtime не блокируется.

3. **Cross-platform:**
   - Каждый backend — отдельный крейт с минимальными платформенными зависимостями.
   - Компиляция на неподдерживаемой платформе безопасно исключает нерелевантный код.
   - Feature flags для выбора backend'а будут добавлены при появлении второй платформы.
