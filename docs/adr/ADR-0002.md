# ADR-0002: Windows UI tools and process management

**Status:** Accepted
**Date:** 2026-07-02
**Deciders:** @kurosss

---

## Context

После внедрения ToolRegistry и ElementSelector (ADR-0001) необходимы конкретные инструменты для автоматизации Windows-интерфейса и управления процессами. Требуется определить:

1. **Паттерн поиска элементов** — как инструменты получают UI-элементы.
2. **Ввод текста** — разница между имитацией ввода (send_keys) и программной установкой (ValuePattern).
3. **Управление процессами** — запуск, остановка и инвентаризация процессов.

---

## Decision 1: FindTool — live search pattern

### Проблема

UI-элементы могут стать невалидными (окно закрылось, элемент пересоздан). Хранить элементы долго — рискованно.

### Решение

**FindTool** выполняет поиск элемента «в моменте» по переданному `ElementSelector` и сохраняет `SafeUIElement` в `ExecutionContext` под указанным ключом.

Все последующие инструменты (ClickTool, InputTextTool, SetTextTool) принимают либо:
- `element_key` — ссылку на элемент в контексте (найден ранее через FindTool), либо
- полный селектор (name, automation_id, control_type, class_name, pid) — для поиска на лету.

```rust
// Пример: поиск + клик
let mut selector = ElementSelector::new()
    .name("Submit")
    .control_type("Button")
    .pid(1234);
let element = selector.find_from_desktop()?;
ctx.set("btn", ContextValue::Custom(Arc::new(SafeUIElement::new(element))));

// Позже — клик
click.execute(json!({ "element_key": "btn" }), ...)

// Или в одну команду:
click.execute(json!({
    "name": "Submit",
    "control_type": "Button",
    "pid": 1234
}), ...)
```

### Consequences

- Элемент ищется при каждом запросе — актуальность гарантирована.
- Если элемент ищется на лету каждый раз — небольшой overhead на UIA-запрос.
- FindTool нужен только для случаев, когда один и тот же элемент используется многократно.
- Все найденные элементы — `SafeUIElement` с `Arc<UIElement>` — живут пока есть хотя бы одна ссылка в контексте.

## Decision 2: InputTextTool — send_keys vs InputTextTool — ValuePattern

### Проблема

Существует два принципиально разных способа ввода текста в UI-элемент:

1. **SendKeys** — имитация нажатия клавиш (медленнее, но триггерит все обработчики).
2. **ValuePattern** — программная установка значения (быстрее, но не все элементы поддерживают).

### Решение

Создаём два отдельных инструмента:

**InputTextTool** (`windows.input_text`):
- Использует `UIElement::send_keys()`.
- Фокусирует элемент перед вводом.
- Подходит для полей ввода, которые реагируют на клавиатурные события.

**SetTextTool** (`windows.set_text`):
- Использует `ValuePattern::set_value()`.
- Работает только с элементами, поддерживающими ValuePattern (Edit, ComboBox).
- Быстрее, не требует фокуса.

### Consequences

- Два инструмента вместо одного с флагом — явнее для LLM (JSON Schema).
- InputTextTool работает везде, SetTextTool — быстрее на поддерживаемых элементах.
- LLM-агент выбирает инструмент под задачу: для быстрого заполнения — SetTextTool, для триггера onChange-обработчиков — InputTextTool.

## Decision 3: ProcessTool — start via std::process::Command, stop/list via taskkill/tasklist

### Проблема

Управление процессами требует:
- Запуска с аргументами и рабочей директорией.
- Остановки по PID или имени процесса.
- Получения списка запущенных процессов.

### Решение

**ProcessTool** (`windows.process`) с тремя действиями:

| Action | Реализация |
|--------|-----------|
| `start` | `std::process::Command::new(...).spawn()`. Возвращает PID. |
| `stop` | `taskkill /F /PID <pid>` или `taskkill /F /IM <name>`. |
| `list` | `tasklist /FO CSV /NH` с парсингом CSV. |

**Почему не WinAPI напрямую:**
- `CreateProcess` / `TerminateProcess` / `EnumProcesses` требуют `windows-sys` и unsafe.
- `std::process::Command` даёт кросс-платформенный запуск.
- `taskkill`/`tasklist` — стабильные утилиты Windows, не требующие FFI.
- Консистентность: до перехода на платформенные вызовы, инструмент уже работает.
- Оптимизация до WinAPI может быть выполнена позже, если потребуется производительность или расширенные возможности.

### Consequences

- Запуск процесса — кросс-платформенный (работает на Linux/macOS тоже, но модуль за `cfg(windows)`).
- Остановка и список — только Windows (через taskkill/tasklist).
- Простота ценой зависимости от внешних утилит (taskkill, tasklist).
- PID, возвращаемый `start`, может быть передан в ElementSelector или SetTextTool/InputTextTool для привязки к процессу.

---

## Consequences (summary)

1. **Live search** — FindTool ищет элемент при каждом запросе, гарантируя актуальность.
2. **Два инструмента ввода** — InputTextTool (send_keys) и SetTextTool (ValuePattern) покрывают все сценарии.
3. **ProcessTool** — запуск через std, stop/list через встроенные утилиты Windows.
4. **PID flow** — ProcessTool → FindTool (по PID) → ClickTool/InputTextTool/SetTextTool — полный lifecycle автоматизации.
